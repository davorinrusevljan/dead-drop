| **Description (short)** | Privacy-focused, ephemeral data sharing with zero-knowledge client-side AES-256-GCM encryption. Share encrypted secrets that self-destruct after 7 days. No account needed. Open source. |
| **Full Description** | dead-drop is a privacy-focused, ephemeral data sharing tool that lets you share encrypted text snippets which automatically self-destruct after 7 days.

How it works: Each drop gets a memorable 4-word name (like "abacus-abide-ablaze-able") instead of a random URL. The name stays in the browser's URL fragment and is never sent to the server — only its SHA-256 hash reaches the backend. You choose a passphrase to encrypt the content client-side using AES-256-GCM with PBKDF2 key derivation (100,000 iterations). The server never sees plaintext. After 7 days, the drop is permanently deleted.

Unlike One-Time Secret, Privnote, or similar tools that give you an unpronounceable URL plus a separate random password (requiring two channels to share), dead-drop gives you a human-readable name. You can tell someone in person or over the phone: "Go to dead-drop.xyz and type abacus-abide-ablaze-able." Both the drop name and your passphrase are speakable — no copy-paste required.

Key features: Zero-knowledge client-side encryption (AES-256-GCM + PBKDF2), auto-destruct after 7 days, version history (up to 5 versions per drop), no account/signup/tracking, public and private drops, public API with OpenAPI spec, open source (MIT). |

### Draft Body:

```
Hi everyone! I've been working on dead-drop, a privacy-focused ephemeral data sharing tool, and I'd love to get your feedback.

**What it does:** Lets you share encrypted text snippets that self-destruct after 7 days. No account needed.

**What makes it different:**

Most tools like this give you a long random URL and a separate password you have to communicate over a different channel. dead-drop takes a different approach — each drop gets a memorable 4-word name (like "abacus-abide-ablaze-able") that's easy to share verbally or over the phone. The name stays in the URL fragment and is never sent to the server — only the SHA-256 hash reaches the backend.

**Key features:**
- Zero-knowledge: Client-side AES-256-GCM encryption with PBKDF2 (100k iterations). Server never sees plaintext.
- Human-readable names via URL fragment — name never leaves the browser
- Ephemeral: drops auto-destruct after 7 days
- Version history: up to 5 versions per drop
- No account, no tracking, no signup
- Public API with OpenAPI spec for programmatic access
- Open source (MIT)

**Tech:** Built on Cloudflare Workers + D1, Hono, Next.js.

It's live at [dead-drop.xyz](https://dead-drop.xyz). Source code is on [GitHub](https://github.com/davorinrusevljan/dead-drop).

Would love to hear what you think — especially about the human-readable names approach and whether the privacy model makes sense to you.
```

## Competitor Analysis

### Direct competitors (secret/paste sharing tools)

| Service | URL format | Human-readable? | Name sent to server? | Account? | Open source | Unique features |
|---------|-----------|-----------------|---------------------|----------|-------------|----------------|
| **dead-drop** | `dead-drop.xyz/#name` | ✅ 4-word Diceware names | ❌ Fragment stays client-side | ❌ Never | ✅ MIT | Version history, public API, memorable names |
| **One-Time Secret** | `onetimesecret.com/secret/abc123` | ❌ Random string | ✅ Yes | Optional | ✅ MIT | Regional domains, most established (2.8k ⭐) |
| **dele.to** | Random ID | ❌ No | ✅ Yes | ❌ No | ✅ MIT | Cleanest UX, "vs PasswordPusher" comparison |
| **Paaster** | Random hash | ❌ No | ✅ Yes | ❌ No | ✅ AGPL-3.0 | Pastebin-style, QR code sharing (534 ⭐) |
| **PrivateBin** | Random hash | ❌ No | ✅ Yes | ❌ No | ✅ Zlib | Most popular (8.2k ⭐), self-hostable |
| **crypt.fyi** | Random ID | ❌ No | ✅ Yes | ❌ No | ✅ Open source | Post-quantum ML-KEM encryption, CLI + Chrome extension |
| **scrt.link** | Random ID | ❌ No | ✅ Yes | ✅ Required | ✅ MIT | File transfer, read receipts |

### dead-drop's unique differentiators

1. **Memorable names via URL fragment** — the ONLY tool where both the drop identifier AND the concept are human-readable and speakable. You can tell someone "dead-drop.xyz, abacus-abide-ablaze-able" in person. No competitor can do this.

2. **Public API with OpenAPI spec** — the ONLY tool with a documented public API for programmatic access. Developer-first approach.

3. **Version history** — up to 5 versions per drop. No competitor offers this.

### Where dead-drop is weaker

- **First-impression UX** — competitors have a single textarea + one button. dead-drop has two panels (Create + View) which looks more complex, even though a name is pre-filled
- **Brand recognition** — PrivateBin (8.2k ⭐), One-Time Secret (2.8k ⭐) are established
- **Post-quantum encryption** — crypt.fyi markets ML-KEM which attracts privacy enthusiasts
- **Self-hostable** — PrivateBin and Paaster can be self-hosted, dead-drop cannot
